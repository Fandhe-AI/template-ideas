# 気をつけるポイント・よくある失敗

## フェーズ共通

### スコープクリープ（スコープの肥大化）
- **症状**: フェーズを進めるたびに要件が増え続ける
- **原因**: 「ついでにこれも」「あれもあったほうが良い」の積み重ね
- **対策**: Must/Should/Could/Won't の分類を厳守する。追加要件は Won't に入れて記録だけする
- **判断基準**: 「これがなくてもユーザーの課題は解決できるか？」→ Yes なら Must ではない

### 完璧主義の罠
- **症状**: ドキュメントが完璧になるまで次のフェーズに進めない
- **原因**: 不確実性への不安
- **対策**: 各フェーズの「完了条件」を満たせば次に進む。後から戻って修正できる
- **心構え**: 80%の完成度で次に進み、必要に応じて戻る方が全体の質は高くなる

### フェーズの飛ばし
- **症状**: ブレストからいきなり実装に入る
- **原因**: 「わかっているから大丈夫」という過信
- **対策**: 各判定ゲートを必ず通過する。飛ばす場合は README.md にスキップ理由を記録する
- **教訓**: 「わかっている」と思っていたことがPoCで覆ることは非常に多い

## Phase 1: ブレインストーミング

### 課題と解決策の混同
- **症状**: 解決策から議論が始まり、本当の課題が見えない
- **例**: 「チャットボットを作りたい」← これは解決策。課題は「カスタマーサポートの応答が遅い」
- **対策**: まず課題（ペイン）を定義し、その後で解決策を考える

### 類似サービスの調査不足
- **症状**: 既に同じものが存在することに後から気づく
- **対策**: ブレスト段階で必ず類似サービスを調査する。差別化ポイントがなければ再考する
- **活用**: Claude Code の Web 検索で類似サービスを網羅的に調べてもらう

### ターゲットが広すぎる
- **症状**: 「すべての開発者向け」のような曖昧なターゲット設定
- **対策**: 最初のユーザーを1人具体的にイメージする
- **効果**: ターゲットが具体的になると、要件も自然と具体的になる

## Phase 2-3: PoC

### PoC不足での見切り発車
- **症状**: 「たぶん大丈夫」でPoCをスキップし、実装後に致命的な問題を発見
- **対策**: 「やってみないとわからないこと」を1つでも残さない
- **判断基準**: 不確実性が残る状態でPhase 4に進まない

### PoCの過剰実施
- **症状**: 確認するまでもないことまでPoCして時間を浪費
- **対策**: 既知の技術で確実にできることはPoC不要。判断に迷ったら poc-planner エージェントに相談
- **目安**: PoC項目が5つを超えたら、優先度が低いものを切り捨てる

### PoC結果の記録不足
- **症状**: 「動いた」とだけ記録して、後から詳細が思い出せない
- **対策**: テンプレートに沿って必ず結果を記録する。特に「発見事項」と「要件への示唆」が重要
- **コツ**: PoCを実施しながらリアルタイムで記録する

### 成功基準の後付け
- **症状**: PoCの結果を見てから「これは成功」と判断する（確証バイアス）
- **対策**: PoC計画時に成功基準を定義する。結果が出てからの変更は禁止
- **例外**: 予想外の良い結果が出た場合は、その旨を記録した上で基準を更新する

## Phase 4: 要件定義

### 曖昧な要件
- **症状**: 「使いやすいUIにする」「高速に動作する」のような要件
- **対策**: 数値や具体的な条件で定義する。「500ms以内にレスポンス」「3クリック以内で目的の画面に到達」
- **テスト**: 受け入れ基準を読んで、テストケースが書けるか確認する

### 非機能要件の見落とし
- **症状**: 機能は定義されているが、パフォーマンス・セキュリティ・運用が考慮されていない
- **対策**: テンプレートのチェックリストを必ず確認する
- **重要項目**: パフォーマンス、セキュリティ、可用性、スケーラビリティ、運用保守性

### 要件の粒度がバラバラ
- **症状**: 「ユーザー認証」（大きい）と「ボタンの色を青にする」（小さい）が同列
- **対策**: 同じ粒度で記述する。大きい要件は分割し、小さい要件は統合する

## Phase 5-6: タスク分解・ロードマップ

### 楽観的すぎる見積もり
- **症状**: すべてが計画通りに進む前提のスケジュール
- **対策**: バッファ（10〜20%）を見込む。初めての技術を使う場合はさらに加算
- **現実**: 見積もりが正確であることは稀。短めより長めに見積もる方が健全

### 依存関係の見落とし
- **症状**: 並行して進めるつもりが、実際には前のタスクが終わらないと始められない
- **対策**: 依存関係マップを必ず作成する。各タスクの「前提タスク」を明示する
- **確認方法**: 「このタスクを今日始められるか？」と自問する

### マイルストーンが大きすぎる
- **症状**: 1つのマイルストーンに多数のタスクが詰め込まれ、完了が見えない
- **対策**: 1マイルストーン = 2〜4週間が目安。それ以上なら分割する
- **原則**: 各マイルストーンにデモ可能な成果物を含める

## 判定ゲートを軽視しない

判定ゲートは面倒に感じるが、最も重要なプラクティスの一つ。

### よくある失敗パターン
1. **ゲートを形骸化する**: 「まあ大丈夫でしょう」で通過 → 後で大きな手戻り
2. **ゲートを厳しくしすぎる**: 完璧を求めて前に進めない → 分析麻痺
3. **ゲートをスキップする**: 「時間がないから」→ 結果的にもっと時間がかかる

### 健全なゲート判定のコツ
- 判定基準は事前に定義する（フェーズの最初に）
- 判定は記録する（日付・結果・理由）
- 迷ったら「Conditional Go」で条件付き続行する
- No-Go は失敗ではなく、適切な判断として評価する
