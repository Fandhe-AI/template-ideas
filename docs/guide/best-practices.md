# ベストプラクティス集

## ブレインストーミングのコツ

### 発散と収束を分ける
- 発散フェーズでは判断を保留する。「それは無理」と思っても記録する
- 収束フェーズで初めて実現可能性を評価する
- 2つのフェーズを混ぜると、アイデアの幅が狭まる

### 「なぜ」を5回繰り返す
- 表面的な課題の裏に本質的な課題がある
- 例: 「レポート作成が遅い」→「なぜ？」→「データ収集に時間がかかる」→「なぜ？」→「データが複数システムに分散している」→ 本質は「データ統合」の問題

### 既存の解決策を調べる
- 完全に新しいアイデアは稀。類似の解決策が必ずある
- 類似サービスの「何が良くて何が足りないか」を分析する
- 差別化ポイントを明確にする

### 「誰のため」を具体的にする
- 「みんなに便利」は要件が定まらない
- 具体的なペルソナを1人決める
- その人の1日の中でどの瞬間に使うかをイメージする

## PoC設計の勘所

### 最小限で検証する
- PoCの目的は「できるか確認する」こと。完成品を作ることではない
- 検証に必要な最小限のコードだけを書く
- UIやエラーハンドリングは後回し

### 仮説を明確に書く
- 悪い例: 「APIが使えるか調べる」
- 良い例: 「OpenAI API のレスポンスが500ms以内に返り、精度90%以上を達成できる」
- 仮説が明確なら、成功/失敗の判断も明確になる

### 成功基準を先に決める
- PoCを始めてから「これは成功と言えるのか？」と迷わないために
- 数値基準があるとベスト（レスポンス時間、精度、コストなど）
- 「動いた」は成功基準ではない

### 失敗を恐れない
- PoCの目的は「やってみないとわからない」ことを検証すること
- 失敗は貴重な情報。「この方法ではダメ」という知見を記録する
- 失敗したPoCの結果も必ずドキュメントに残す

## 要件定義のポイント

### ユーザーストーリーで書く
- 「ログイン機能」ではなく「ユーザーとして、安全にログインしたい。なぜならアカウントの不正利用を防ぎたいからだ」
- 「なぜなら」の部分が要件の本質を明確にする
- 「なぜなら」が書けない要件は、本当に必要か再考する

### 受け入れ基準をテスト可能にする
- 悪い例: 「高速に動作すること」
- 良い例: 「1000件のデータに対して3秒以内にレスポンスを返すこと」
- 受け入れ基準がそのまま受け入れテストになる

### Must要件を厳選する
- Must要件が多すぎるとMVPが肥大化する
- 目安: Must要件は全体の30〜40%
- 「Must要件だけでプロダクトとして成立するか？」が判断基準

### 除外事項を明示する
- 「やらないこと」を書くのは「やること」と同じくらい重要
- スコープクリープを防ぐ最も効果的な手段
- 「将来やるかもしれないがv1では対象外」も記録する

## タスク分解のコツ

### 1タスク = 1成果物
- 成果物が明確でないタスクは分解が不十分
- 「調査する」よりも「調査結果をドキュメントにまとめる」
- 「実装する」よりも「〇〇APIのエンドポイントを実装する」

### 依存関係を最小化する
- 並行して進められるタスクが多いほど効率的
- 依存関係が多い場合は、インターフェースを先に定義するタスクを作る
- ボトルネック（多くのタスクが依存するタスク）を最優先にする

### Claude Code と人間の役割分担
- **Claude Code が得意**: コード生成、テンプレート作成、リファクタリング、テスト作成、ドキュメント生成
- **人間が判断すべき**: ビジネス判断、UI/UXデザイン、外部との調整、最終的な品質承認
- **共同作業**: 設計議論、コードレビュー、要件の詳細化

## ロードマップ作成のポイント

### MVPを最初のマイルストーンに
- Must要件だけで構成するMVPを最初に完成させる
- 早くフィードバックを得られる
- 万が一スケジュールが遅れても、MVPは手に入る

### バッファを見込む
- 見積もりの10〜20%をバッファとして加える
- 不確実性が高い部分はさらにバッファを多くする
- バッファは「怠けるため」ではなく「想定外に対応するため」

### レビューポイントを設ける
- マイルストーンごとに「続行するか」を判断するタイミングを設ける
- 市場環境やビジネス状況の変化に対応できる
- レビューポイントで方向転換する勇気を持つ
