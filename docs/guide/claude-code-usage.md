# Claude Code の効果的な活用法

## エージェントの使い分け

### 各エージェントの特性

| エージェント | 最適な場面 | 使い方のコツ |
|-------------|-----------|-------------|
| `brainstorm` | アイデアの初期段階、自由な発想が欲しいとき | 答えを求めず問いかけに答える形で進める |
| `poc-planner` | 技術的リスクを洗い出したいとき | ブレスト結果を事前に完成させておく |
| `requirements` | 要件を体系的に整理したいとき | PoC結果を全て読んでもらう |
| `task-decomposer` | 実装計画を立てたいとき | 要件の優先度が決まっている状態で使う |
| `roadmap` | スケジュールを策定したいとき | タスク分解が完了している状態で使う |

### エージェント vs スキル

- **エージェント**: 対話しながら深く掘り下げたいときに使う（例: ブレスト、要件の詳細化）
- **スキル**: 定型的な操作を素早く行いたいときに使う（例: 新しいアイデアの作成、ステータス確認）

---

## スキルの活用

### `/new-idea` — 新しいアイデアを始める
```
/new-idea my-awesome-tool
```
- ディレクトリ作成からブレスト開始まで一気に進む
- 名前は kebab-case で指定する
- 日本語のアイデア名は README.md 内に記載する

### `/advance-phase` — 次のフェーズに進む
```
/advance-phase my-awesome-tool
```
- 現フェーズの完了チェックが自動で行われる
- 判定ゲートがある場合は判断を求められる
- 次フェーズのエージェントが自動起動する

### `/idea-status` — 進捗確認
```
/idea-status               # 全アイデア一覧
/idea-status my-awesome-tool  # 特定アイデアの詳細
```

### `/idea-summary` — サマリー生成
```
/idea-summary my-awesome-tool
```
- 全フェーズのドキュメントを読み込んで要約する
- 他の人に共有するときや、しばらく間が空いた後の復帰時に便利

## 効果的なプロンプトの書き方

### ブレスト段階

**効果的なプロンプト例:**
```
こういう課題を感じている: [課題の説明]
こんな解決策を考えた: [ざっくりしたアイデア]
壁打ち相手になって、このアイデアを深掘りしてほしい
```

**避けるべきプロンプト:**
```
いいアイデアを考えて  ← 主体がClaude側になってしまう
```

### PoC段階

**効果的なプロンプト例:**
```
以下の仮説を検証するPoCコードを書いてほしい:
- 仮説: [具体的な仮説]
- 検証方法: [どう確認するか]
- 成功基準: [具体的な数値]
```

**避けるべきプロンプト:**
```
とりあえず動くものを作って  ← 何が検証できたかわからなくなる
```

### 要件定義段階

**効果的なプロンプト例:**
```
03-poc/ の全結果を踏まえて、04-requirements.md を作成してほしい。
特に以下の点に注意:
- PoC-2 で判明した制約を反映する
- 非機能要件もカバーする
```

## Claude Code との協働のコツ

### 1. コンテキストを共有する
- Claude Code は会話の中で共有された情報しか知らない
- 新しいセッションを始めるときは、`/idea-summary` で背景を共有する
- 判断の背景にある「なぜ」を伝えると、より適切な提案が得られる

### 2. 判断は人間がする
- Claude Code はオプションの提示と分析が得意
- 最終的な判断（Go/No-Go、優先度、スコープ）は人間が行う
- 「どちらがいいか？」より「AとBのメリット・デメリットを比較して」の方が有用

### 3. 段階的に進める
- 一度に全部を頼まない
- フェーズごとに、対話しながら進める
- 途中で方向性を修正できる余地を残す

### 4. テンプレートを活用する
- テンプレートに沿って進めると漏れが少ない
- テンプレートが合わない場合は、テンプレート自体を改善する
- カスタマイズは歓迎。ただし必須項目は残す

### 5. 記録を怠らない
- 対話の中で出た重要な決定は必ずドキュメントに記録する
- 「なぜその判断をしたか」の理由も記録する
- 将来の自分や他のメンバーへの贈り物になる

## トラブルシューティング

### エージェントが期待通りに動かない
- CLAUDE.md の内容が正しく読み込まれているか確認する
- エージェント定義ファイルの構文エラーがないか確認する
- 直接プロンプトで指示してみて、エージェントの定義と比較する

### フェーズの進行が滞る
- 完了条件を確認して、何が不足しているか特定する
- 不足している部分に集中して取り組む
- 完璧を目指さず、80%の完成度で次に進む

### アイデアの方向性を見失った
- `/idea-summary` で全体を振り返る
- `01-brainstorm.md` に立ち返って「そもそも何を解決したかったか」を確認する
- 必要なら Phase 1 からやり直す（手戻りは健全なプロセス）
