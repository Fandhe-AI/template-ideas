# 各フェーズの詳細ガイド

## Phase 1: ブレインストーミング

### 目的
アイデアの種を膨らませ、解決すべき課題と解決策の方向性を明確にする。

### ゴール
- 課題が具体的に言語化されている
- 解決策の方向性が1〜3案に絞られている
- ターゲットユーザーが明確になっている
- 暫定的な要件リストがある

### 進め方
1. **発散フェーズ**: アイデアを自由に出す。否定しない。
2. **深掘りフェーズ**: 「なぜ？」を繰り返して本質的な課題に到達する。
3. **調査フェーズ**: 類似サービスや既存の手法を調べる。
4. **収束フェーズ**: アイデアを構造化し、要件概要としてまとめる。

### Claude Code の活用
- `brainstorm` エージェントが壁打ち相手になる
- 類似サービスの調査を Web 検索で支援
- 散らばった議論を構造化してドキュメントに整理

### 完了条件
- `01-brainstorm.md` が作成されている
- 要件概要（暫定）が少なくとも3項目以上ある
- 未解決の疑問点が整理されている

### 判定ゲート: 追求判定
以下を考慮して判断する:
- このアイデアの実現可能性は？
- 投資するリソースに見合う価値があるか？
- 致命的なブロッカーはないか？

## Phase 2: PoC計画

### 目的
ブレインストーミング結果から不確実性の高い部分を特定し、検証計画を立てる。

### ゴール
- 検証すべき項目が優先度付きで一覧化されている
- 各項目の仮説・検証方法・成功基準が明確
- 実施順序が依存関係を考慮して決まっている

### 進め方
1. 01-brainstorm.md の要件概要と未解決の疑問を確認
2. 「やってみないとわからないこと」をリストアップ
3. 各項目に仮説・検証方法・成功基準を定義
4. 優先度と実施順序を決定

### PoC対象の判断基準
| PoC対象 | PoC不要 |
|---------|---------|
| 技術的に実現可能か不明 | 既知の技術で確実に実現可能 |
| パフォーマンス要件を満たせるか不明 | 軽微な機能で失敗リスクが低い |
| 外部サービスとの連携が未検証 | 社内で実績のある手法 |
| ユーザー受容性が不明 | 要件が明確で議論の余地がない |
| コストが大きく失敗が許されない | 小さく試して修正できる |

### 完了条件
- `02-poc-plan.md` が作成されている
- 各PoC項目に仮説・検証方法・成功基準がある
- Go/No-Go の全体判断基準が定義されている

---

## Phase 3: PoC実施

### 目的
計画に基づいて各PoC項目を検証し、結果を記録する。

### ゴール
- 全PoC項目の検証が完了している
- 各項目の結果が成功基準と照合されている
- 要件への示唆が整理されている

### 進め方
1. `02-poc-plan.md` の実施順序に従ってPoC項目を進める
2. 各PoC項目ごとに `03-poc/<poc-item>/` ディレクトリを作成
3. コードや設定ファイルなどの成果物を配置
4. `README.md` に結果を記録（テンプレート: `_templates/03-poc-result.md`）
5. 全項目完了後、Go/No-Go判定を行う

### Claude Code の活用
- PoC のコード実装を支援
- 結果の分析と整理
- 成功基準との照合

### 完了条件
- 全PoC項目のディレクトリと結果ドキュメントがある
- 各項目の判定（OK/NG/条件付きOK）が記録されている

### 判定ゲート: Go/No-Go判定（最重要）
- **Go**: 全検証項目の成功基準を満たす
- **Conditional Go**: 一部妥協可能な項目がある（条件を明記）
- **No-Go**: 致命的な問題が発見された（Phase 1に戻って再検討 or 中止）

## Phase 4: 要件定義

### 目的
PoC結果を踏まえて、実装に必要な詳細要件を策定する。

### ゴール
- 機能要件がユーザーストーリー形式で記述されている
- 非機能要件が具体的な数値基準で定義されている
- MoSCoW優先度が付けられている
- スコープの境界が明確

### 進め方
1. 全PoC結果を踏まえて要件を整理
2. 各要件をユーザーストーリー形式で記述
3. 受け入れ基準をテスト可能な形で定義
4. MoSCoW分類で優先度を付ける
5. 除外事項を明確にする

### MoSCoW分類
| 区分 | 基準 | 例 |
|------|------|-----|
| Must | なければプロジェクトが成立しない | コア機能 |
| Should | 重要だが回避策がある | UX改善 |
| Could | あると良いが必須ではない | 追加機能 |
| Won't | 今回は対象外 | 将来のアイデア |

### 完了条件
- `04-requirements.md` が作成されている
- Must要件が3つ以上定義されている
- 各要件に受け入れ基準がある

### 判定ゲート: スコープ確認
- 要件の総量は現実的か？
- Must要件だけでMVPが成立するか？
- 不足している要件はないか？（追加PoCが必要なら Phase 2-3 へ）

## Phase 5: タスク分解

### 目的
要件を実装可能な粒度のタスクに落とし込む。

### ゴール
- 各要件に対する具体的なタスクが定義されている
- タスク間の依存関係が整理されている
- 工数の見積もりがある
- Claude Code と人間の担当区分が明確

### 進め方
1. 各要件を1〜2日で完了できるタスクに分解
2. タスク間の依存関係を特定
3. 各タスクに工数見積もりを付ける
4. 担当区分（Claude Code / 人間 / 共同）を決定
5. 優先順位を付ける

### タスク粒度の目安
| 粒度 | 判断 |
|------|------|
| 0.5〜2日 | 適切 |
| 3日以上 | さらに分解する |
| 数時間以下 | 他のタスクと統合を検討 |

### 完了条件
- `05-tasks.md` が作成されている
- 全Must要件に対応するタスクがある
- 依存関係マップがある
- 見積もりサマリーがある

---

## Phase 6: ロードマップ

### 目的
タスクをマイルストーンにまとめ、実現可能なスケジュールを作成する。

### ゴール
- マイルストーンが定義されている
- 各マイルストーンにデモ可能な成果物がある
- リスクと対策が整理されている
- レビューポイントが設定されている

### 進め方
1. タスクをマイルストーンにグルーピング
2. 依存関係を考慮してタイムラインを作成
3. リスクと対策を整理
4. レビューポイント（Go/No-Go含む）を設定
5. バッファ（10〜20%）を見込む

### マイルストーン設計の原則
- 各マイルストーンにデモ可能な成果物を含める
- Must要件を最初のマイルストーンに集中させる
- 各マイルストーン完了時にGo/No-Go判定を設ける

### 完了条件
- `06-roadmap.md` が作成されている
- 3つ以上のマイルストーンがある
- リスクと対策が整理されている

### 判定ゲート: 着手判定
- ロードマップは現実的か？
- リソースは確保できるか？
- 実装を開始する準備が整っているか？
