# ワークフロー全体像

## 6フェーズ＋判定ゲート

```
Phase 1: ブレスト
  │  アイデアの発散、課題の深掘り、要件概要の固め
  │
  ▼ [追求判定] このアイデアにPoCの価値があるか？
  │  → No: アイデアを保留 or 破棄
  │
Phase 2: PoC計画
  │  技術的リスクと不確実性の特定、検証項目の洗い出し
  │
  ▼
Phase 3: PoC実施
  │  各検証項目の実施、結果の記録
  │
  ▼ [Go/No-Go判定] PoC結果を踏まえて先に進めるか？ ← 最重要ゲート
  │  → No-Go: Phase 1に戻って再検討 or 中止
  │  → Conditional Go: スコープを絞って続行
  │
Phase 4: 要件定義
  │  PoC結果を踏まえた詳細要件、MoSCoW優先度付け
  │
  ▼ [スコープ確認] 要件の範囲は妥当か？
  │  → 過大: スコープを絞る
  │  → 不足: 追加PoCが必要なら Phase 2-3 に戻る
  │
Phase 5: タスク分解
  │  要件ごとの具体タスク、工数見積もり、依存関係
  │
  ▼
Phase 6: ロードマップ
  │  マイルストーン、タイムライン、リスク管理
  │
  ▼ [着手判定] 実装を開始するか？
```

## フィードバックループ

ワークフローは一方通行ではありません。以下のケースでは前のフェーズに戻ります:

| 状況 | 戻り先 | 例 |
|------|--------|-----|
| PoC失敗・想定外の制約発見 | Phase 1 | アプローチの根本的な見直し |
| 要件定義中にPoC不足を発見 | Phase 2-3 | 追加の技術検証が必要 |
| タスク分解で複雑すぎると判明 | Phase 4 | スコープの縮小 |
| ロードマップが非現実的 | Phase 4-5 | 要件やタスクの見直し |

戻る場合、README.md のステータスを「🔁 再検討中」に更新します。

## ステータス管理

各アイデアの `README.md` にステータステーブルがあります:

| 記号 | 意味 |
|------|------|
| ⬜ | 未着手 |
| 🔄 | 進行中 |
| ✅ | 完了 |
| 🔁 | 再検討中 |
| ⏸️ | 保留 |
| ❌ | 中止 |

## 判定ゲートの考え方

判定ゲートは「次に進んでいいか」を意識的に判断するポイントです。

### なぜ判定ゲートが必要か

- **サンクコスト回避**: 早い段階で「やらない」判断ができる
- **品質保証**: 不十分な状態で次に進むことを防ぐ
- **リソース管理**: 投資すべきアイデアにリソースを集中できる

### 判定の記録

判定結果は該当フェーズのドキュメント末尾に記録します:

```markdown
## 判定
- **結果**: Go / No-Go / Conditional Go
- **理由**: ...
- **条件**:（Conditional Go の場合）...
- **日付**: YYYY-MM-DD
```
